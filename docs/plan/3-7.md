# Harness vs OpenAI Codex — Gap Analysis & Roadmap (2026-03-07)

## Overall Assessment: ~35-40% Feature Completeness

Previous estimate of 50-60% was revised downward after deeper comparison revealed OpenAI's significant advantages in cloud execution, OS-level sandboxing, SDK, and CI/CD integration.

## Priority-Ranked Key Gaps

### P0 — Architectural Gaps (Must-have to be competitive)

#### 1. Real Agent Loop

OpenAI's core is a prompt->tool->observe->iterate loop. Harness only does single subprocess invocation.

- Current: `CodeAgent::execute()` spawns `claude -p` or `codex exec`, waits for exit, returns full output
- Target: Iterative loop with tool execution, observation, and re-prompting within a single turn
- Impact: Without this, harness cannot support mid-turn steering, tool approval, or incremental output
- Files: `harness-agents/src/claude.rs`, `harness-agents/src/codex.rs`, `harness-core/src/agent.rs`

#### 2. Protocol Compatibility

- Method naming: `snake_case` -> slash style (`thread/start`, `turn/start`)
- Initialize handshake: mandatory `initialize` -> `initialized` sequence
- Bidirectional notifications with filtering
- Files: `harness-protocol/src/methods.rs`, `harness-server/src/router.rs`

#### 3. Streaming Delta Output

- Turn/item level realtime push: `turn/started`, `item/started`, message deltas
- Currently execute_stream() runs full execution then sends one ItemCompleted
- Files: `harness-agents/src/claude.rs`, `harness-protocol/src/notifications.rs`

#### 4. OS-Level Sandbox

- macOS: Seatbelt policies via `sandbox-exec`
- Linux: Landlock + seccomp; alternative bwrap pipeline
- Three modes: read-only, workspace-write (default), danger-full-access
- Protected paths: `.git`, `.agents/`, `.codex/` always read-only
- Files: New crate `harness-sandbox/`

### P1 — Capability Gaps (High ROI improvements)

#### 5. GitHub Webhook -> Task Automation

- `@harness` mention on issue/PR triggers task creation
- Auto-review on new PRs
- CI failure detection and auto-fix
- Files: `harness-server/src/http.rs` (new webhook endpoint)

#### 6. AGENTS.md Support

- Cascading discovery: global -> project root -> subdirectories
- Override mechanism: `AGENTS.override.md` replaces guidance at that level
- 32KB combined limit, configurable
- Inject into agent prompts automatically
- Files: `harness-core/src/prompts.rs`, new `agents_md.rs`

#### 7. Approval Policy System

- Three levels: suggest (default), auto-edit, full-auto
- Granular reject map: `{ sandbox_approvals: true, mcp_elicitations: true }`
- Admin enforcement via requirements.toml
- Files: `harness-core/src/config.rs`, `harness-server/src/task_executor.rs`

#### 8. CI/CD GitHub Action

- Trigger on PR events (opened, synchronize, reopened)
- Install harness CLI, run `harness exec`
- Security: drop-sudo, unprivileged-user, allow-users/bots
- Output: final-message for PR comments
- Files: New `.github/actions/harness-action/`

#### 9. Multi-Agent Dispatch Activation

- Wire `AgentRegistry::dispatch()` into main execution path
- Register codex + anthropic-api at startup
- Complexity-based routing (simple -> default, complex -> claude)
- Files: `harness-server/src/http.rs`, `harness-server/src/task_runner.rs`

### P2 — Nice-to-Have

#### 10. TypeScript/Python SDK

- `startThread()`, `thread.run(prompt)`, event streaming
- Mirror `@openai/codex-sdk` API surface

#### 11. Cloud Execution

- Container isolation per task
- Two-phase: setup (network on) -> agent (network off)
- Secrets management: available during setup, removed before agent phase
- Container caching (up to 12 hours)

#### 12. MCP Server Mode

- `harness mcp-server` exposes `harness` and `harness-reply` tools
- External orchestrators (Agents SDK, Claude Code) can control harness programmatically

#### 13. Starlark Rules Engine

- Replace shell guard scripts with Starlark (sandboxed Python-like)
- `prefix_rule()` for command matching with smart shell parsing
- `codex execpolicy check` equivalent for rule testing

#### 14. OpenTelemetry Export

- Native OTLP export (traces, metrics, logs)
- `otlp-http` and `otlp-grpc` exporters
- Config: `[otel] environment, exporter, log_user_prompt`

## Harness Unique Advantages (Not in OpenAI Codex)

## Recommended 3-Step Roadmap

### Step 1: Agent Loop + Streaming (Highest Impact)

Transform subprocess single-call into real prompt->tool->observe loop with streaming delta output. This is the architectural foundation everything else depends on.

- Refactor `CodeAgent::execute()` to support iterative tool calls
- Implement turn-level streaming (item/started, message deltas, item/completed)
- Wire WebSocket `/ws` route for HTTP server push
- Estimated scope: ~2000 lines across harness-agents, harness-server, harness-protocol

### Step 2: Protocol Alignment + AGENTS.md

Make harness compatible with OpenAI Codex clients and support project-level instructions.

- Slash-style method names with backward compatibility layer
- Initialize handshake (`initialize` -> `initialized`)
- AGENTS.md cascading discovery and prompt injection
- Estimated scope: ~800 lines across harness-protocol, harness-server, harness-core

### Step 3: GitHub Automation + Approval Policies

Enable team deployment with webhook-driven task creation and human-in-the-loop controls.

- GitHub webhook endpoint for `@harness` mentions
- Auto-review on PR creation
- Three-level approval policy (suggest/auto-edit/full-auto)
- Estimated scope: ~1200 lines across harness-server, harness-core
