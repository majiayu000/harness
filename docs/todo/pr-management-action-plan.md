# PR Management Action Plan

> Generated by multi-AI cross-validation (2026-04-12)
> Sources: 3 internal agents + Grok (Gemini unavailable)

## Action Items (tiered by confidence)

### 🔴 Strong Consensus — 立即执行

#### 1. Feature PR 不再修改 Cargo.toml version
**依据**: tokio, deno, tauri, rust-lang, OpenAI codex 全部这样做。Grok 称 release-plz "事实标准"。
**操作**:
- 删除 CLAUDE.md 中 "每次提交都必须更新 Cargo.toml 中的版本号" 规则
- 删除 `~/.claude/CLAUDE.md` 中相同规则
- 版本号只在发布时统一 bump
**预期**: 消除 90%+ 的 Cargo.toml 合并冲突
**Rollback**: 恢复 CLAUDE.md 规则即可

#### 2. 切换为 Squash-merge-only
**依据**: OpenAI codex (200+ 并发 PR) 强制 squash-only。3/4 agent + Grok 一致推荐。
**操作**:
```
GitHub Settings → Repo → General → Pull Requests:
  ☑ Allow squash merging (default)
  ☐ Allow merge commits
  ☐ Allow rebase merging
```
**预期**: 每个 PR = 1 commit，大幅减少冲突面
**Rollback**: 重新开启其他 merge 方式

#### 3. Model Tiering（简单任务用 Haiku）
**依据**: Agent C 发现 ReasoningBudget 基础设施已存在。Grok 称 "Haiku 处理 80% 日常任务，成本仅 1/3"。
**操作**:
- 在 config/default.toml 中为 rebase/format/version-bump 任务配置 Haiku
- 仅 implement + 复杂 review 用 Sonnet
- 对应修改 task_executor.rs 中的 model 选择逻辑
**预期**: 节省 40-60% Sonnet token 消耗
**关键文件**:
- `crates/harness-core/src/types.rs` (ReasoningBudget, L588-655)
- `crates/harness-core/src/config/agents.rs` (ClaudeAgentConfig, L227)
- `crates/harness-server/src/task_executor.rs` (model selection per phase)

#### 4. 减少 Review 轮次
**依据**: 3 agent + Grok 一致。OpenAI "agent-to-agent review" 也验证了短循环有效。
**操作**:
- 全局设 `concurrency.max_turns = 20`（防失控）
- Medium 复杂度从 8 轮降到 5 轮
- Rebase/简单任务 cap 在 3 轮
**预期**: 节省 20-30% token
**关键文件**: config/default.toml, task_runner.rs

### 🟡 Partial Consensus — 建议做

#### 5. Path-based CI skip
**依据**: OpenAI codex 实测有效（agent 确认）。Grok 推荐。
**操作**: 在 `.github/workflows/ci.yml` 中添加 changed-files 检测，跳过无关 crate 的测试
**前提**: CI 中有多个独立的 test job
**预期**: PR CI 从 ~10min 降到 ~3-5min

#### 6. Stale PR 自动关闭
**依据**: OpenAI codex 用 14 天自动关闭。防止 PR 堆积。
**操作**: 添加 `.github/workflows/close-stale-prs.yml`
**前提**: 需要确认不会误关正在进行的工作

#### 7. Quota-aware Throttling
**依据**: Agent C 发现 `wait_for_rate_limit()` hook 点已存在。Grok 推荐。
**操作**: 扩展 `wait_for_rate_limit` 检查 token_usage 数据，低于 30% 时降低并发
**前提**: 需要 token_usage 模块从只读变为可执行检查
**关键文件**: task_runner.rs L1678

#### 8. 集成 release-plz
**依据**: Agent B + Grok 强推荐。自动生成 Release PR + CHANGELOG。
**操作**: 添加 `release-plz.toml` + GitHub Action
**前提**: 需要决定发布节奏（weekly/per-milestone）

### ⚪ 待验证 — 需要更多数据

#### 9. 是否需要 Merge Queue？
**矛盾**: OpenAI codex 200+ PR 不用 merge queue（Agent A 实测）。Grok 推荐 Mergify。
**仲裁**: OpenAI 靠 squash-merge + fast CI + stale cleanup 替代。Harness 只有 18 PR，更不需要。
**结论**: 先实施 1-4，观察效果。如果仍有问题，再考虑 GitHub native merge queue。

### 🚫 不做

#### ~~Mergify / Aviator~~
**反对依据**: OpenAI 200+ 并发 PR 不用任何 merge queue 工具。Harness 规模远小于此。付费工具引入不必要的依赖。

#### ~~Bors~~
**反对依据**: Grok 明确说 "2023 年后已弃用"。仅 rust-lang/rust 还在用自定义版本。
